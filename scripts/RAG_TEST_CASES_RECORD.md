**用途**：記錄 AI 辨識結果與實際物種的對照，供後續分析使用。  
**狀態**：收集中，暫不修改程式或提出建議。

---

## 測試環境說明

- **拍攝方式**：在螢幕上拍攝照片
- **流程**：於網站上找到植物照片 → 開啟照片 → 用手機對準螢幕拍攝（圈選後上傳）
- **原因**：無法外出逐株實地拍攝，植物種類眾多，故以螢幕呈現的網路照片作為測試素材
- **說明**：照片來源為螢幕翻拍，非直接實地拍攝植物

## 演算法參考（供後續分析納入考量）

- **兩階段比較門檻**：第二階段最高分須 > 第一階段最高分 + **0.15**（15%）才替換，否則保留第一階段
- **PLANT_SCORE_THRESHOLD**：0.4（Vision 判斷為植物）～ 0.5（其他情況），未達門檻不進行 RAG
- **混合搜尋權重**：embedding 60%、feature 40%
- **LM 加成**：LM 名稱與 RAG 結果匹配時，給予約 40% 信心度加成
- **詳細文件**：`SCORING_ALGORITHM_DOCUMENTATION.md`

## 預期 vs 實際：AI 特徵提取範圍

- **當初設計**：希望 AI 辨識並輸出「果實」「根莖」「花色」等多項特徵
- **目前觀察**：多以「花序」為主，較少完整描述果實、根莖、花色等特徵
- **測試案例中的現象**：
  - flower_color 常被誤提取為「紅種子」「白種子」「黃種子」「紫葉」（案例 1、3、4、5、9、10、12、14）
  - 花序類型（inflorescence）較常出現，但 racemose、spiral、pinnate 等部分值無中文映射被忽略（案例 3、7、12）
  - 果實型態、根莖特徵在 query_text_zh 中較少見
- **花色 vs 葉色混淆**：無花朵時，AI 常將「有顏色的葉片」誤判為「花朵」
  - 例：紫葉、紅葉植物被識別為「紫花」「紅花」等
  - 影響：flower_color 填入錯誤，導致 RAG 特徵匹配偏離
- **影響**：特徵提取偏重葉片＋花序，果實、根莖、花色等可區分物種的特徵較弱，且花色／葉色易混淆，可能降低 RAG 匹配與辨識準確度

## 向量資料污染可能性（final-4302 檢視）

- **重複中文名**：存在多筆同中文名不同來源（例：七里香 4 筆、仙丹花 2 筆、大冇樹 4 筆等），同一物種多向量可能稀釋或汙染 embedding 空間
- **高頻通用特徵**：互生 1509 次、灌木 1097、喬木 1094、全緣 514、卵形 452、總狀 422……當查詢為「灌木、互生、披針形、鋸齒」等通用詞時，易命中大量植物，embedding 難以區分
- **錦帶花／美蕊花／仙丹花**：描述多含「灌木、互生/對生、卵形、紅花、繖房/總狀」等通用組合，可能成為 embedding 空間中的「聚類中心」，易被當成多種查詢的 top 結果
- **結論**：污染風險存在（重複名、通用特徵過度集中），需納入判斷因素

## 補充討論變因（供演算法改善方向）

| 變因 | 說明 | 與辨識率的關聯 |
|------|------|----------------|
| **螢幕翻拍畫質** | 非實地拍攝，解析度、色彩、反光可能影響 Vision 特徵提取 | 特徵錯誤 → traits 錯誤 → RAG 匹配偏離 |
| **TraitsParser 映射缺失** | racemose、spiral、pinnate、pinnately_compound 等無中文映射被忽略 | 關鍵特徵遺失，第二階段 query 不完整 |
| **兩階段 15% 門檻** | 第二階段須高 15% 才替換，門檻過高易使正確結果無法取代第一階段 | 案例 3 金雀花第二階段第一、但因分數低未替換 |
| **embedding 權重偏高** | embedding 60%、feature 40%，通用詞易拉高 embedding 分數 | 特徵匹配無法有效校正通用 embedding |
| **負向特徵／排除邏輯** | 無「絕對不符合則排除」機制（例：羽狀複葉 vs 單葉） | 形態差異大的物種仍可能被召回 |
| **候選數量** | top_k=3，若正確答案不在前 3 則無法選中 | 可考慮增加候選或調整篩選策略 |
| **資料庫覆蓋** | 巴西野牡丹、念珠珊瑚等不在庫內 | 無法辨識屬資料集限制 |

**目標**：將上述變因轉化為演算法調整，提升辨識率至約 70%。

## 第二階段問題點分析（依目前 3 筆再測）

- **現象**：只有風鈴草一筆出現 `results 未定義`（其餘兩筆第二階段正常）。
- **可能原因**：
  1. **後端收到空 body 或 body 不完整**  
     Node 有送 `query=42字 features=12 guess_names=30`，但若 Zeabur/代理或網路在該次請求把 body 截斷或丟失，Python 端會讀到 `query=""`、`features=[]`、`guess_names=[]`，進而回傳 **400** `Missing 'query', 'features', or 'guess_names'`，回應裡只有 `error`、沒有 `results` → Node 就會得到 `results 未定義`。
  2. **該次請求逾時或連線中斷**  
     若請求在送出後或回傳前逾時/中斷，Node 可能收到不完整或非 JSON 回應，`result.data` 結構異常，也會沒有 `results`。
  3. **偶發性**  
     同一風鈴草若再測有時正常、有時異常，就較可能是網路/代理或負載造成的偶發問題。
- **建議**：
  - 到 **embedding-api（Zeabur）日誌** 查該次風鈴草請求：是否有 `POST /hybrid-search 收到: query_len=42...`。  
    - 若**沒有**這行、而是 `400: 缺少 query/features/guess_names (body keys: [])` → 可判斷為 **body 未送達或為空**（代理/Content-Length/逾時等）。  
    - 若有 `收到: query_len=42` 但之後出現 `encode_text 失敗` 或 `Qdrant 回傳 0 筆` → 問題在後端邏輯或 Qdrant/Jina，而非 body。
  - 若日誌顯示 body 有收到且無錯誤，但 Node 仍拿到無 `results` 的回應，再檢查是否有 **回應被截斷**（例如逾時關閉連線、回應過大被截斷等）。
